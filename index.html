자료구조와 알고리즘의 이해

자료구조와 알고리즘의 이해

1. 자료구조

  • 자료구조 ≒ 데이터 구조 ≒ data structure
  • 대량의 데이터를 효율적으로 관리할 수 있는 데이터의 구조
  • 어떤 데이터 구조를 사용하느냐에 따라 코드 효율이 달라지므로, 데이터 특성에 따라 체계적으로 데이터를 구조화해야 함

2. 효율적인 데이터 관리의 예

  • zipcode : 5자리 우편번호로 국가의 기초구역을 제공 (시군구 코드 3자리 + 일련번호 2자리)
  • 학생 관리 : 학년 / 반 / 번

3. 대표적인 자료 구조

  • 개발 : 배열, 스택, 큐, 링크드 리스트, 해쉬 테이블, 힙 etc
    현실 : 사전

4. 알고리즘 (algorithm)

  • 어떤 문제를 풀기 위한 절차/방법
  • 어떤 문제에 대해 특정한 ‘입력’을 넣으면 원하는 ‘출력’을 얻을 수 있도록 만드는 프로그래밍
    현실 : 백종원 레시피

5. 자료구조와 알고리즘의 중요성

  • 어떤 자료구조와 알고리즘을 쓰는지에 따라 성능 차이가 크게 남
  • 프로그래밍의 기술과 역량을 검증할 수 있음

6. 자료구조/알고리즘 and 파이썬

  • C/C++ 에서 언어에 대한 제약과 평가가 없어지는 추세
  • python : 가장 쉽고 빠르게 자료구조와 알고리즘을 익힐 수 있음 / 현업 사용도가 높음

7. 기본 환경

  • 컴파일러 : 프로그래밍 언어로 작성된 코드 -> 컴퓨터가 실행할 수 있는 코드로 변환해주는 프로그램
  • 라이브러리 : 소프트웨어 개발에 쓰이는 하부 프로그램들의 모임
    pip install library-name
  • 에디터 / IDE : jupyter notebook, pycharm

V. Tread

  • Thread : 프로그램 내부의 흐름, 맥(脈)
  • 일반적으로 하나의 프로그램은 하나의 스레드를 가진다

1. 다중 스레드(Multi Thread)

  1. 다중스레도

    • thread가 빠른 간격으로 스위칭되며 여러개가 동시에 실행되는 것 처럼 보임 (concurrent)
      cf.simultaneous : 진짜로 동시에 2개 이상이 작동하는 것

  2. 다중 스레드 예시

    • Web browser : 화면 출력하는 스레드 + 데이터 읽어오는 스레드
    • Word processor : 화면 출력하는 스레드 + 키보드 입력 받는 스레드 + 철자/문법 오류 확인 스레드
    • 그 외 음악 재생, 동영상 플레이어, Eclipse IDE 등 대부분의 프로그램이 멀티스레드가 구현됨

  3. Thread vs Process

    • 한 프로세스에는 하나(혹은 그 이상)의 스레드가 존재함
    • context switching 단위는 이전에는 프로세스였으나 현재는 스레드임
    • 스레드는 프로세스의 메모리(code, data)와 자원(file, i/o) 공유
    • Program Counter, register, stack 등은 공유하지 않음 > 스위칭 될 때마다 값이 바뀌므로

운영체제 4

III. 프로세스 관리

1. 프로세스

  1. 프로세스 : program in execution
  2. 프로세스 상태 및 천이
    • new / ready / running / waiting / terminated
    • ready > CPU > running
    • running > I/O start > waiting
    • waiting > I/O end > ready
    • running > Time Expired > ready (Time-Sharing System)

2. PCB (Process Control Block)

  • TCB(Tack Control Block)
  • 프로세스에 대한 모든 상태 정보가 담겨 있음
    process state, PC(program counter, 순서), register, MMU(base, limit), CPU Time, id, list of open files …
  • OS의 프로세스 관리 부서에 위치함

3. Queue{: name = ‘Queue’}

  1. Job Queue (long-term scheduler, 메모리가 빌때만 작동하므로 시간이 많이 소요됨)
    • 프로세서에 의해서 처리되기를 기다리고 있는 job의 집단이나 대기 행렬
    • Job Scheduler : Job Queue의 우선순위 결정, OS의 프로세스 관리 부서
  2. Ready Queue (short-term scheduler)
    • Ready 상태에 있는 프로세스들, 즉 메모리에 load되어있는 queue
    • CPU Scheduler
  3. Device Queue
    • device를 사용하기 위해 대기하는 queue
    • Device Scheduler

4. Multiprogramming

  1. Degree of multiprogramming
    • 메인 메모리에 프로세스를 얼마나 많이 올릴 수 있는가?
  2. I/O-bound(입출력) process vs CPU-bound(계산)
    • process 를 적절히 믹스해 job scheduling > 컴퓨터의 성능 향상
  3. mediun-term scheduler
    • 유휴 메모리를 main memory에서 swap device(backing store, HDD)로 swap in/out 하여 메모리 공간 확보
  4. Context Switching(프로세서의 프로세스 전환)
    • Scheduler : Ready Queue에 의한 switching
    • Dispatcher : 프로세서가 스케쥴링 된 실행 순서로 CPU에 올리는 것
      dispatch : ready > running
      컨텍스트의 현재 상태 저장, 불러오기 등의 업무를 수행
    • Context Switching Overhead
      프로세스 전환할 때 발생하는 부담
    • 어셈블리 같은 효율이 높은 low level 언어로 개발함

IV. CPU 스케쥴링

  • ready queue에 대기한 프로세스들을 어떤 식으로 처리할지에 대한 방법
  • 대기시간을 기준으로 계산

1. CPU Scheduling

  1. Preemptive(선점) VS Non-Preemptive(비선점)
    • 응급환자(우선순위 역전) VS 일반환자 진료(우선순위 순)
  2. Scheduling Criteria (스케쥴링 기준)
    • CPU Utilization (이용률, %)
    • Throughput (단위 시간 당 처리율, jobs/sec)
    • Turnaround time (반환 시간, ready > running > terminated, sec)
    • Waiting time (대기 시간, ready > running, sec)
    • Response time (응답 시간, I/O response, 대화형 컴퓨터)

2. CPU Scheduling Algorithms

  1. FCFS(First-come, first-served)
    • 단순, 공정
    • 평균 대기 시간의 함정에 빠질 수 있음
    • Convoy Effect(호위 효과) : 실행 시간이 오래 걸리는 작업이 앞에 위치해 다른 작업들이 줄줄이 뒤따르는 현상
    • 비선점 스케쥴링
  2. SJF(Shortest-Job-First)
    • 실행시간이 가장 짧은 작업을 우선 처리
    • 평균 대기시간을 단축할 수 있음 (증명 완료)
    • 단점 : 실제 실행 시간이 얼마나 걸릴지 정확히 알 수 없음, 이전의 작업을 통해 실행시간을 예측해서 측정해야 함
    • 선점, 비선점 모두 가능
      Non-preemptive : p1 > p2 > p4 > p3 = (0 + (8-1) + (17-2) + (12-3))/4 = 7.75 ms
preemptive : p1(1) > p2(4) > p4(5) > p1(7) > p3(9) = ((0 + 9) + 0 + (17 - 2) + (5 - 3))/4 = 6.5 ms // 최소잔여시간 우선 shortest-remaining-time-first, 실행 중 작업 추가되면 최소잔여시간 비교하여 선점 작업 교체
  3. Priority Scheduling
    • 내부 요소 : time limit, memory requirement, CPU burst
    • 외부 요소 : 비즈니스, 정치적 요인
    • 선점, 비선점 모두 가능
    • 우선순위에 밀려 대기시간이 길어지는 starvation(기아) 현상 발생
    • 단점 해결 : aging, 대기시간을 우선순위 선정에 반영함
    • p2 > p5 > p1 > p3 > p4 = (6 + 0 + 16 + 18 + 1) / 5 = 8.2 ms
  4. Round-Robin
    • Time Sharing System
    • Time Quantum(∆)은 보통 10~100ms, time quantum에 따라 성능이 결정됨
    • 하지만 time quantum을 0으로 수렴하면 잦은 context switching으로 overhead가 발생함
    • Only Preeemptive
    • (Time Quantum : 4ms) p1 > p2 > p3 > p1 > p1 > p1 … = (0 + 3 + 3) + 4 + (4 + 3) = 17 / 3 = 5.66 ms
  5. Multilevel Queue Scheduling
    • Process Group : 목적이나 용도에 따라 프로세스를 그룹핑(시스템, 인터렉티브, 배치 등)하여 여러개의 queue를 만들어 독립된 스케쥴링 정책 설정 가능
    • 각 queue에 절대적 우선순위를 부여하여 cpu time을 차등 배분함
      보통 system > interactive > batch > etc.. 순으로 우선순위 결정
  6. Multilevel Feedback Queue Scheduling
    • 작업이 지연될 경우 다른 queue로 이동

3. 프로세스의 생성과 종료

  1. 프로세스 생성

    • (init) > 부모 > 자식 프로세스 - 형제(slbling) 프로세스

    • 프로세스 트리를 그릴 수 있음

    • PID : unique한 int 부여, PPID : 부모의 PID

    • fork() : 부모 프로세스를 복사하는 시스템 콜 github fork

    • exec() : 실행 파일을 메모리로 가져옴 .exe파일

    • exit() : 프로세스 종료, 프로세스 실행 시 가졌던 리소스(메모리, 파일, I/O 등)는 모두 OS에 반환

II. Interrupt-Based System

1. 인터럽트

  1. 인터럽트(중단, 새치기)
    • 프로세서(CPU)가 프로그램을 실행하고 있을 때, 입출력 하드웨어 등의 장치나 또는 예외상황이 발생하여 처리가 필요할 경우에 프로세서에게 알려 처리할 수 있도록 하는 것
    • 현대 운영체제는 인터럽트 기반 시스템
    • 부팅이 끝난 후 운영체제는 메모리에 상주(resident)하며 사건(event)을 기다리며 대기
  2. 하드웨어 인터럽트
    • 인터럽트 결과 운영체제 내의 특정 코드(Interrupt Service Routine) 실행
    • 실행 종료 후 대기
    • 키보드 마우스 입력 등의 동작
  3. 소프트웨어 인터럽트(swi)
    • 애플리케이션이 실행중 다른 프로그램을 실행 시 소프트웨어 인터럽트 발생(예: hwp 이용 중 하드에서 파일을 읽어오는 경우)
    • ISR 실행, 종료 후 원래 프로그램으로 복귀
  4. 내부 인터럽트(Internal Interrupt)
    • divide by zero

2. 이중모드

  1. 한(1) 컴퓨터를 여러 사람(N)이 동시에 사용하는 것
    • 여러 개의 프로그램을 한 사람이 사용
    • 한 사람의 고의 혹은 실수 프로그램이 전체에 영향
  2. 사용자 프로그램은 STOP 등의 치명적 명령에 접근 불가하도록 설정
    • User Vs Supervisor(System, Monitor, Priviliged)
  3. 특권 명령(priviliged instructions)
    • STOP, HALT, RESET, SET_TIMER, SET_HW …
  4. CPU = register(bit) + ALU(연산장치) + ControlUnit
    • register : carry, negative, zero, overflow 등의 모드(flag 0/1)로 표현
    • register에 이중모드를 나타내는 bit를 추가
  5. 이중모드 작동 process
    • Power On -> Booting (System Mode 1)
    • Application, User Program (System Mode 1인 상태에서 프로그램 메모리에 로딩 > System Mode 0 > User Mode)
    • 인터럽트 (ex : Application 의 Data를 HDD에 저장, 프린트 등), System Mode 1 > User Mode
    • 즉 하나의 프로그램이 실행될 때 이중모드의 상태가 계속 변경된다.
    • User Mode에서 강제로 특권 명령 실행 시도할 경우
      권한 없는 사용자가 잘못된 명령을 내렸다는 내부 인터럽트 발생하여 ISR에 의해프로그램 강제 종료

3. 보호(Protection)

  1. 입출력장치 보호
    • 키보드, 마우스, 프린터, 랜, 스피커, 모니터 등
    • IN / OUT 명령어를 특권 명령으로 설정
    • User Mode에서 명령 접속시 priviliged instruction violation으로 인식하여 ISR 실행하여 프로그램 강제 종료
    • solution : 운영체제에 요청하여 System Mode로 전환 후 작업 > 작업 마치면 User Mode 복귀, 올바른 요청이 아니면 거부
  2. 메모리 보호
    • 다른 사용자의 메모리, 운영체제 영역 메모리 접근 (Hacking)
    • Segment Violation 발생
    • CPU -> [ MMU (BASE~LIMIT) ] -> MEMORY : Address Bus
      잘못 접근할 시 ISR 실행
    • Memory Management Unit의 BASE, LIMIT 값 설정은 특권명령
  3. CPU 보호
    • 한 사용자가 실수 혹은 고의로 CPU 시간 독점 (ex : 재귀 함수)
    • solution : Timer를 두어 일정시간 경과 시 타이머 인터럽트
      인터럽트 > OS > 다른 프로그램으로 강제 전환

4. 운영체제 서비스 (OS = Government)

  1. 프로세스 관리(Process Management)
    • 프로세스 : 메모리에서 실행 중인 프로그램(program in execution)
    • 프로세스 생성, 소멸(creation, deletion)
    • 활동 일시 중지, 재개(suspend, resume)
    • 프로세스간 통신(IPC:interprocess communication)
    • 동기화(synchronization)
    • 교착상태 처리(deadlock handling)
  2. 주기억장치 관리(Main Memory Management)
    • 프로세스에 메모리 할당 (allocation)
    • 할당된 메모리 추적, 감시
    • 프로세스 종료 시 메모리 회수
    • 메모리의 효과적 사용
    • 가상메모리 : 실제 메모리보다 큰 용량의 메모리 갖도록
  3. 파일 관리(File Management)
    • Track/Sector - ‘파일’ Mapping
    • 파일/디렉토리의 생성, 삭제 및 기본 동작 (open, close, read, write, create, delete)
    • 백업
  4. 보조기억장치 관리(Secondary Storage Management)
    • HDD, Flash Memory etc
    • 빈 공간(block) 관리, 할당, 스케쥴링
  5. 입출력 장치 관리(I/O Device Management)
    • 하드웨어 장치 드라이브
    • buffering, caching, spooling등의 방법으로 입출력 장치 성능 향상

5. 시스템 콜(System Call)

  • 프로그램 > 운영체제 서비스를 받기 위한 호출
  • Process: end, abort, load, execute, create, terminate, get/set
    attributes, wait event, signal event
  • Memory: allocate, free
  • File: create, delete, open, close, read, write, get/set attributes
  • Device: request, release, read, write, get/set attributes,
    attach/detache devices
  • Information: get/set time, get/set system data
  • Communication: socket, send, receive

I. OS Introduction

1. 운영체제란?

  1. Windows 10, 7, Linux, MS-DOS, Mac OX, iOS
  2. 운영체제 : Operating System
  3. 컴퓨터에 운영체제가 없다면?
    • 컴퓨터: 프로세서(CPU) & 메모리
    • 전원을 켜면 휘발성 메모리(RAM)가 먼저 활성화됨 -> 야생마와 같은 상태
  4. 운영체제의 역할
    • 단일 프로그램 실행
    • 프로그램을 여러개 실행
    • 인쇄, 출력, 저장
  5. 운영체제의 의의
    • 하드웨어 관리
    • 성능 향상 (Performance)
    • 편의성 (Convinence)
    • Control program for Computer

2. Booting

  1. 컴퓨터 구조
    • 프로세서(CPU), 메모리(RAM, ROM), 디스크
    • POST(Power-On Self-Test)
    • 부트 로더(Boot Loader)
  2. 부팅 과정
    • 컴퓨터 Power On > POST(Power-On Self-Test) >Rom (Bootstrap Load) > HDD 안의 OS > 메인메모리(RAM)로 가져옴
  3. 운영체제 구성
    • 관리(Management) 프로그램
    • 프로세서, 메모리, 디스크, 입출력장치 드라이브
    • 커널(Kernal) vs 명령 해석기(Shell, Command Interpreter)
  4. 운영체제의 위상
    • 하드웨어 > 운영체제(커널 > 쉘) > 애플리케이션
  5. OS vs Government
    • 자원 관리자(resource manager)
    • 자원 할당자(resource allocator)
    • 주어진 자원의 활용
    • 직접 일하지 않음
    • 업무별로 부서가 할당됨

3. OS History : 하드웨어 발전 / 운영체제 발전

  1. 시초 : 1940년대 말
    • card reader > memory > processing > line printer
  2. Batch Processing System(일괄처리) : Resident Monitor
  3. Multiprogramming System(다중프로그래밍)
    • 아직까지 컴퓨터는 비싼 자원
    • 빠른 COU vs 느린 i/o -> 메모리에 여러개의 job over loading
    • CPU 스케쥴링, 메모리 관리, 보호가 중요
  4. Tims-Sharing System(시공유 시스템)
    • ms 단위로 강제 전환, interactive system(대화형)
    • 가상 메모리, 프로세스간 통신, 동기화

4. 고등 운영체제

  1. 다중 프로세서(Multiprocessor System)
    • 병렬(parallel)
    • 강결합(tightly-coupled)
    • performance, cost, reliability의 장점이 있음
    • 1 memory, multi CPU
  2. 분산 시스템(Distributed System)
    • 다중 컴퓨터 시스템(multi-computer)
    • 소결합(loosely-coupled)
    • 분산 운영체제(Distributed OS)
    • multi cpu, multi memory, connected by lan
  3. 실시간 시스템(Real-time System)
    • 시간 제약(deadline)이 있는 프로그래밍
    • 공장 자동화(FA), 군사, 공항, 우주, 네비게이션, etc
    • 실시간 운영체제(RTOS)
